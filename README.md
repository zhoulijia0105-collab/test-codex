# 招聘决策诊断小工具（MVP）

一个本地可运行的最小版本网页工具，用于把“要不要招人”先转化为结构化诊断：
- 先判断问题本质
- 再判断是否建议招聘
- 输出可解释理由与下一步建议

> 第一版采用**规则引擎**，不依赖外部 API，不接数据库。

---

## 1. 技术栈选择

- **后端：Python + Flask**
  - 依赖少、启动快、学习成本低。
  - 很适合做本地 MVP 和规则引擎 API。
- **前端：HTML + 原生 JS + CSS**
  - 无前端构建工具，新手可直接理解与改造。
- **数据层：无数据库**
  - 规则直接在代码中维护，便于第一版快速迭代。

## 3. 本地启动

### 环境要求
- Python 3.10+

### 安装与运行


## 4. 使用方式

1. 在输入框写入业务问题（1 句话或一段）。
2. 点击“开始诊断”。
3. 页面会输出 JSON 结构结果，例如：

```json
{
  "问题本质": "当前问题更像“职责缺口/承接缺口”，存在真实人力缺位。",
  "是否建议招聘": "建议",
  "问题类型": "招聘问题",
  "判断理由": [
    "识别到职责缺失或新增业务无人承接信号：没人接、负责人",
    "该结论来自可解释规则引擎（关键词命中与类别得分）。"
  ],
  "建议": [
    "先明确该岗位的结果责任与边界，再启动招聘。",
    "把“必须新增”的工作与“可优化/可外包”的工作拆开。",
    "设定30-60-90天目标，避免“招了但没人管”。"
  ]
}
```

## 5. 规则引擎说明（第一版）

核心逻辑位于 `app/rules.py`：
- 内置 5 类问题：
  - 招聘问题
  - 组织结构问题
  - 流程问题
  - 角色设计问题
  - 管理能力问题
- 每类包含：关键词、问题本质、招聘建议、理由模板、下一步建议。
- 通过关键词命中计分，选择得分最高的类别。
- 若命中不足或并列冲突，返回“信息不足 / 暂不确定”。

这个结构便于后续扩展：
- 新增类别：在 `RULES` 增加配置即可。
- 调整策略：可替换为“关键词+权重”“规则树”“模型评分”等。

---

## 6. 下一步升级到大模型，改哪一层？

建议只替换 **诊断层（rules.py）**，保持前后端接口不变。

- 保留 `POST /api/diagnose` 输出结构不变。
- 在 `diagnose()` 内部增加两阶段策略：
  1. **规则预判**（低成本、可解释、稳定）
  2. **LLM 补充判断**（处理复杂语义、给更细建议）
- 当模型不可用时自动回退规则引擎，保证可用性。

这样升级成本最低，且不影响页面与调用方式。
